1   Abstract

TrainAssist is a cross-platform mobile application developed using the Flutter framework for the front-end and ASP.NET Core Minimal API with a PostgreSQL relational database for the back-end. The application addresses a real and daily challenge faced by millions of train passengers: the lack of real-time, accessible information about coach crowd levels, emergency reporting, lost-and-found coordination, and on-journey situational awareness.

The system allows passengers to search for trains by source and destination, view the crowd occupancy status (Low / Medium / High) of individual coaches as reported by fellow passengers, and submit their own crowd updates. In emergency situations, users can trigger a full SOS flow that captures GPS location, records a voice note, and dispatches alerts via SMS, WhatsApp, and Email (SMTP) to pre-configured emergency contacts. Additional features include a Bluetooth-based station proximity alert, a lost-and-found reporting module, a medical emergency profile accessible from the lock screen, and a rule-based on-device chatbot capable of answering common passenger queries such as live train tracking, PNR status, ticket cancellation, and nearest station directions.

The application was built and deployed as a debug APK on a physical Android 14 (API 34) device — Samsung Galaxy S23 — over a Wi-Fi ADB connection. The entire stack, excluding cloud deployment, is free of cost to run locally.

1.1   Introduction

Rail transport continues to be one of the primary modes of long-distance and inter-city travel across much of the world, particularly in countries like India where the railway network spans thousands of kilometres and serves hundreds of millions of passengers annually. Despite this scale, the average passenger still lacks a simple, unified digital tool to get actionable real-time information about their journey — from coach crowd conditions to emergency assistance.

TrainAssist was conceived to bridge this gap. The application puts several safety and convenience features directly in the pocket of every train passenger. By allowing passengers themselves to report crowd conditions, the system creates a community-driven, real-time occupancy map. The SOS module provides a multi-channel emergency alert system that operates even in partially connected environments, leveraging SMS and WhatsApp which work over cellular data. The chatbot immediately answers common questions without requiring an internet connection.

The project follows an N-tier architecture with a clear separation between:

  • Presentation Layer — Flutter (Dart) mobile app targeting Android.
  • Business / API Layer — ASP.NET Core Minimal API (.NET 10) exposing REST endpoints.
  • Data Layer — Entity Framework Core 10 ORM connected to a PostgreSQL 15 database.
  • On-device Intelligence — Rule-based chatbot (SimpleChatBot) running entirely offline.

Development adopted an iterative, prompt-driven workflow assisted by GitHub Copilot, accelerating feature delivery across multiple sprint-like cycles. Each prompt document (e.g., PROMPT_2_3_COMPLETE.md, PROMPT_7_8_COMPLETE.md) captures the complete implementation of a set of features.

1.2   Brief Overview of Technology

Flutter (Dart)

Flutter is Google's open-source UI toolkit for building natively compiled applications for mobile, web, and desktop from a single codebase in the Dart programming language. The framework's widget-based rendering engine provides pixel-perfect UI on Android and iOS without requiring platform bridges for most components. TrainAssist uses Flutter 3.x / Dart SDK ^3.8.1 and targets Android (primary) and Chrome web (secondary).

ASP.NET Core Minimal API (.NET 10)

Microsoft's .NET 10 platform provides a lightweight, high-performance HTTP API framework. Minimal APIs use a functional, top-level statement style to define endpoints directly in Program.cs without controllers, reducing boilerplate. Entity Framework Core 10 with Npgsql driver handles all database interactions using code-first migrations.

PostgreSQL 15

PostgreSQL is a powerful, open-source object-relational database system known for ACID compliance, rich data types, and mature tooling. The TrainAssist schema includes five tables: Trains, Coaches, CrowdReports, SOSReports, and Users. A DataSeeder class pre-populates the database with sample train and coach data on first run.

Provider (State Management)

The Flutter app uses the Provider package (ChangeNotifier pattern) for state management across eight providers: UserProvider, TrainProvider, CoachProvider, SOSProvider, BluetoothProvider, StationAlertProvider, LostFoundProvider, and MedicalProfileProvider.

2   Existing System

Several applications and platforms currently offer partial solutions for train passengers. An analysis of the existing landscape reveals significant gaps that TrainAssist aims to address:

National Train Enquiry System (NTES – ntes.in)

NTES provides official live running status for all Indian Railways trains. However, it is a web portal without a companion mobile application offering integrated safety features. It does not provide crowd reporting, SOS alerts, or offline capabilities.

RailYatri / ixigo

These third-party platforms offer train status, PNR checking, and ticket booking. Neither provides a community-driven real-time coach occupancy system or a one-tap SOS emergency alert that integrates GPS, voice recording, SMS, and WhatsApp simultaneously.

IRCTC Official App

IRCTC is the official ticketing platform. Its scope is limited to booking and cancellation; it does not address on-journey safety, crowd conditions, or emergency response.

General Limitations of Existing Systems

  • No real-time community-driven coach crowd reporting.
  • No integrated multi-channel SOS (SMS + WhatsApp + Email + Voice Note).
  • No Bluetooth-based station proximity and wake-up alert.
  • No on-device offline chatbot answering train-related queries.
  • No lost-and-found coordination within the journey context.
  • No medical emergency profile stored on the device for first-responders to access.
  • No single app combining all the above in a unified, privacy-first, open-source stack.

Disadvantages of the Existing System

  1. Fragmented experience — passengers must switch between multiple apps.
  2. Most features require internet connectivity at all times.
  3. No personalised emergency configuration (contacts, SMTP, WhatsApp).
  4. SOS communication relies on phone calls only; no automated multi-channel dispatch.
  5. Privacy concerns with large commercial platforms handling location data.

3   Proposed System

TrainAssist proposes a unified, open-source mobile application that consolidates all passenger safety and convenience features under a single interface, with a lightweight REST API backend and an on-device chatbot that works offline.

Key Features of the Proposed System

3.1 Train Search and Coach Crowd Status
  • Search trains by source and/or destination station name (partial match supported).
  • View all coaches for a train with the latest crowd status: Low 🟢, Medium 🟡, or High 🔴.
  • Submit a new crowd report for any coach — persisted in PostgreSQL via the REST API.
  • Crowd data is timestamped; the most recent report is always displayed.
3.2 SOS Emergency Alert System
  • One-tap SOS trigger, or triple-tap screen gesture, or Volume-UP ×3 (Guard Mode).
  • Captures GPS coordinates with reverse geocoding to a human-readable address.
  • Records a voice note (M4A/WAV/Opus) and uploads it to the backend's /api/uploads endpoint.
  • Sends the SOS record (name, train, coach, GPS, message) to the backend /api/sos.
  • Dispatches SMS to all saved emergency contacts using the native SMS composer.
  • Opens WhatsApp with a prefilled SOS message for each contact (optional auto-send via Accessibility service).
  • Sends an email with the voice note attached via SMTP (configurable host, port, credentials).
3.3 Bluetooth Station Proximity Alert
  • User configures a target station name and scans for nearby Bluetooth beacons.
  • When a beacon matching the station name is detected, a high-priority full-screen notification is fired.
  • Prevents passengers from sleeping through their stop on long-distance journeys.
3.4 Lost and Found
  • Passengers can report lost or found items specifying description, location, and contact info.
  • The app automatically matches lost reports with found reports and notifies both parties.
  • Entirely local/on-device storage (SharedPreferences); no backend dependency.
3.5 Medical Emergency Profile
  • Users enter their blood group, allergies, medical conditions, and emergency contact.
  • A persistent, high-priority lock-screen notification (via flutter_local_notifications) keeps the medical card accessible at all times — critical when a passenger is unconscious.
3.6 On-Device Chatbot
  • Rule-based bot (SimpleChatBot) runs entirely offline — no AI API costs.
  • Handles: live train tracking links, ETA, PNR status, ticket cancellation/refund, nearest station, SOS flow, helpline numbers (139 / 182), crowd reporting guide, WhatsApp setup, and SMTP configuration.
  • Returns tappable links opening the appropriate page in the browser or Google Maps.
3.7 REST API and Swagger
  • ASP.NET Core Minimal API exposes fully documented endpoints at /swagger.
  • Endpoints: GET /api/trains, GET /api/trains/{id}/coaches, POST /api/coaches/{id}/crowd, POST /api/sos, GET /api/sos, POST /api/uploads, POST /api/sendmail, GET /api/trains/{num}/links, GET /api/health.

Advantages of the Proposed System

  1. Single unified app replacing five or more separate tools.
  2. Community-driven real-time crowd data with no external API dependency.
  3. Multi-channel SOS that works independently of phone call availability.
  4. Offline chatbot answers common questions without internet.
  5. Open-source, self-hosted — no subscription or usage fees.
  6. Medical card always visible on lock screen — aids first-responders.
  7. Bluetooth station alert eliminates missed-stop anxiety on overnight trains.

7   Summary

TrainAssist is a feature-rich, production-quality mobile application that addresses the real daily needs of train passengers through a carefully chosen technology stack. The project demonstrates:

  • Full-stack development — Flutter/Dart mobile app + ASP.NET Core REST API + PostgreSQL.
  • State management — Provider pattern with eight domain-specific ChangeNotifier classes.
  • Sensor integration — GPS (geolocator), Bluetooth (flutter_blue_plus), microphone (record).
  • Multi-channel communication — SMS, WhatsApp (url_launcher), Email (SMTP via backend).
  • Offline-first design — Chatbot, medical profile, and lost/found operate without internet.
  • Security — CORS configured for development; input sanitised before DB writes; no SQL injection via EF Core parameterised queries; audio uploads restricted to known MIME types.
  • Community data — Crowd reports aggregate passenger feedback into real-time coach occupancy.
  • AI-assisted development — GitHub Copilot accelerated feature development across all modules.

The application was successfully built, installed, and tested on a physical Android 14 device via wireless ADB. All major features — train search, crowd reporting, SOS dispatch (SMS, WhatsApp, email), Bluetooth station alert, lost-and-found, medical profile, and the chatbot — are functional.

8   Conclusion

The TrainAssist project successfully demonstrates how a small, focused mobile application can meaningfully improve the travel experience and safety of train passengers. By combining community-sourced data (crowd reports), device sensors (GPS, Bluetooth, microphone), multiple communication channels (SMS, WhatsApp, email), and an offline-capable intelligent assistant (rule-based chatbot), the application provides value that no single existing platform currently offers.

The choice of Flutter ensures a single Dart codebase that targets both Android and web without duplication of effort. The ASP.NET Core Minimal API provides a lightweight yet production-capable backend that supports Swagger documentation out-of-the-box. PostgreSQL offers a robust, free, and well-understood data store with strong ACID guarantees.

The iterative, AI-assisted development workflow — using GitHub Copilot with structured prompt documents — proved highly effective in accelerating feature delivery while maintaining code quality and architectural consistency. Each major feature set was delivered as a discrete, reviewable increment tracked in dedicated completion documents.

In conclusion, TrainAssist meets and exceeds its initial requirements. It is ready for further development toward public release, with a clear roadmap of enhancements outlined in the next section.

9   Future Enhancement

  1. Real-time push notifications — Integrate Firebase Cloud Messaging (FCM) to push crowd alerts and SOS acknowledgements to devices without requiring the app to be open.
  2. AI-powered chatbot upgrade — Replace the rule-based SimpleChatBot with a small on-device ML model (e.g., TensorFlow Lite) or integrate with an LLM API to handle free-form natural language queries beyond keyword matching.
  3. Live train GPS tracking — Subscribe to official API feeds (NTES/RailYatri) or crowd-source GPS pings from passengers to render a live train position on a map within the app.
  4. Authentication and user accounts — Add OAuth 2.0 / JWT-based authentication so crowd reports and SOS records are tied to verified users, enabling gamification and accountability.
  5. iOS support — Complete the iOS configuration (runner signing, permission plist entries) to publish on the Apple App Store alongside Android.
  6. Cloud deployment — Deploy the ASP.NET Core API and PostgreSQL to a managed cloud platform (Azure / AWS / Railway.app) so the app functions for users across the country without a local server.
  7. Offline crowd data caching — Cache the latest crowd reports locally so users can view coach occupancy even without an internet connection.
  8. Wearable integration — Extend SOS trigger to paired smartwatches (using Wear OS APIs) for even faster emergency dispatch.
  9. Accessibility features — Add voice-over support, large-text mode, and high-contrast themes for visually impaired passengers.
  10. Multi-language support — Add Hindi, Tamil, Telugu, and other regional language localisation using Flutter's intl and ARB-based localisation system.
